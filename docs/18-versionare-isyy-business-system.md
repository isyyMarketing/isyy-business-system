# Versionare ISYY Business System

Versionarea explică evoluția documentației și a sistemului de lucru.

Scopul nu este să documentăm la infinit, ci să știm când sistemul este suficient de clar pentru execuție reală.

## Ce înseamnă versiunile

### v1.0 — Baza sistemului

v1.0 definește:

- viziunea;
- rolurile;
- principiile;
- modurile de lucru;
- workflow-ul;
- stack-ul oficial.

### v1.1 — Șabloane și reguli practice

v1.1 adaugă:

- template-uri pentru Codex;
- template-uri de analiză business;
- template-uri pentru decizii;
- reguli de securitate;
- workflow GitHub;
- organizarea fișierelor.

### v1.2 — Operaționalizare

v1.2 adaugă:

- pași pentru primul proiect;
- reguli `.env` și secrete;
- checklist înainte de commit;
- checklist înainte de lansare;
- exemple completate.

### v1.3 — Structură pentru proiect real

v1.3 adaugă:

- structura standard `projects/_template/`;
- reguli de copiere pentru proiecte noi;
- reguli de versionare;
- reguli pentru branch-uri și Pull Request.

## Când se creează o versiune nouă

Se creează o versiune nouă când:

- apare o regulă importantă nouă;
- se schimbă procesul de lucru;
- se adaugă o structură reutilizabilă;
- se clarifică un risc important;
- sistemul devine mai ușor de folosit.

Nu se creează o versiune nouă pentru modificări mici de text.

## Când se oprește documentarea

Documentarea se oprește temporar când:

- există structură suficientă;
- primul proiect poate fi pornit;
- template-urile sunt clare;
- riscurile principale sunt acoperite;
- următorul pas este execuția.

## Când se trece la execuție reală

Se trece la execuție reală când:

- există un proiect ales;
- folderul proiectului a fost creat din `projects/_template/`;
- ideea este completată;
- analiza business este completată;
- MVP-ul este definit;
- specificația Codex este clară.

Regula simplă: documentația trebuie să susțină execuția, nu să o înlocuiască.
