# Cum pornești primul proiect

Acest document explică pașii concreți pentru pornirea unui proiect nou în ISYY Business System.

Scopul este să treci de la idee la lansare într-un mod clar, fără haos.

## 1. Scrie ideea simplu

Începe cu o descriere scurtă:

- ce vrei să construiești;
- ce problemă rezolvă;
- pentru cine este;
- de ce ar conta.

Nu începe direct cu implementarea.

## 2. Analizează ideea

Folosește template-ul:

```text
templates/business-idea-analysis-template.md
```

Verifică:

- clientul țintă;
- problema rezolvată;
- piața;
- modelul de venit;
- costurile;
- riscurile;
- potențialul de scalare.

La final, decide: continuă, modifică sau oprește.

## 3. Definește MVP-ul

Dacă ideea merită continuată, definește MVP-ul.

MVP înseamnă cea mai simplă versiune utilă care poate testa ideea.

Folosește template-ul:

```text
templates/mvp-validation-template.md
```

Clarifică:

- funcțiile minime;
- ce nu intră în MVP;
- criteriile de validare;
- criteriile de lansare;
- condițiile de oprire.

## 4. Creează specificația pentru Codex

După ce MVP-ul este clar, pregătește o cerere pentru Codex.

Folosește template-ul:

```text
templates/codex-spec-template.md
```

Specificația trebuie să spună:

- ce trebuie implementat;
- ce fișiere pot fi modificate;
- ce fișiere sunt interzise;
- ce nu trebuie schimbat;
- cum se verifică rezultatul.

Codex nu trebuie să primească cereri vagi.

## 5. Implementează pe etape

Codex implementează doar ce este clar stabilit.

Pentru primul proiect, lucrează în pași mici:

1. structură de bază;
2. funcție minimă;
3. documentație;
4. testare locală;
5. ajustări.

Nu adăuga funcții avansate înainte ca MVP-ul să fie validat.

## 6. Testează rezultatul

Isac testează rezultatul ca Product Owner.

Verifică:

- dacă proiectul pornește;
- dacă funcția principală merge;
- dacă mesajul este clar;
- dacă utilizatorul înțelege valoarea;
- dacă există erori evidente.

Problemele găsite se documentează și se rezolvă pe etape.

## 7. Pregătește lansarea

Înainte de lansare, folosește checklist-ul:

```text
docs/16-checklist-inainte-de-lansare.md
```

Verifică:

- MVP-ul;
- bug-urile critice;
- datele sensibile;
- documentația;
- backup-ul;
- costurile;
- metoda de monetizare.

Lansarea se face doar după decizia finală a lui Isac.

## 8. Documentează ce ai învățat

După testare sau lansare, notează:

- ce a mers bine;
- ce nu a mers;
- ce trebuie schimbat;
- ce feedback ai primit;
- care este următorul pas.

Un proiect bun nu este doar construit. Este învățat, ajustat și îmbunătățit.
